## §3 领域规律（≥ 深度门槛 条，每条五要素）

### 规律：主键必须用 IDENTITY/序列，禁止 max(id)+1 应用层取号
- **适用版本**: 全版本（IDENTITY 自 PG 10 引入，SERIAL 为旧式等价物）
- **规律**: `SELECT max(id)+1` 在并发下必产生重复主键（两个事务读到同一 max），且 max 扫描随表增大变慢。必须用 `GENERATED ALWAYS AS IDENTITY`（SQL 标准，禁止手插）或 `GENERATED BY DEFAULT AS IDENTITY`，旧项目 SERIAL 可保留。序列取号有间隙（回滚不回收）属正常，禁止为"连续编号"改用 max 模式。
- **违反后果**: 并发重复主键 → 写入报错/数据错乱；大表 max 扫描拖垮插入路径。
- **验证方法**: `grep -rnEi 'max[[:space:]]*\([[:space:]]*(id|[a-z_]+_id)[[:space:]]*\)[[:space:]]*\+' --include='*.sql'` → fail。
- **对应门禁**: fw_pgsql_pk_identity(fail)

### 规律：JSON 列必须用 JSONB（二进制、可索引），禁用 json 类型
- **适用版本**: 全版本
- **规律**: `json` 类型原样存文本，每次查询重新解析、不支持索引；`jsonb` 存解析后的二进制，支持 GIN 索引、`@>` 包含查询、键去重。除非必须保留原始格式/键顺序（审计原文场景），一律 `jsonb`。
- **违反后果**: json 列查询全解析全表扫；无法加索引，JSON 检索接口随数据量劣化。
- **验证方法**: DDL 检出列类型 `json`（词边界排除 jsonb）→ warn。
- **对应门禁**: fw_pgsql_json_vs_jsonb(warn)

### 规律：JSONB 列有检索必须建 GIN 索引
- **适用版本**: 全版本
- **规律**: JSONB 的 `@>` / `?` / `?|` / `?&` 操作符只有 GIN 索引能加速（`USING gin (payload)` 或 `jsonb_path_ops`）。建了 jsonb 列但不建 GIN 索引 = 检索全表扫。写多读少且从不按 JSON 内容查询的列可不建（规律按"schema 检出 jsonb 即提示核对"执行）。
- **违反后果**: JSON 检索全表顺序扫描 → 大表接口超时。
- **验证方法**: schema 文件含 `jsonb` 但同文件无 `USING gin`/`USING GIN` → warn 核对。
- **对应门禁**: fw_pgsql_jsonb_index(warn)

### 规律：autovacuum 不可关闭，防表膨胀与事务 ID 回卷
- **适用版本**: 全版本
- **规律**: MVCC 的 UPDATE/DELETE 产生死元组，靠 VACUUM 回收；`autovacuum=off` 导致表/索引持续膨胀、统计信息失真，极端时事务 ID 回卷（xid wraparound）迫使数据库只读停机。高更新表应调低 `autovacuum_vacuum_scale_factor` 而非关闭；大促前手工 `VACUUM (ANALYZE)`。
- **违反后果**: 表膨胀 10 倍 + 查询劣化；xid 回卷 → 库级只读事故。
- **验证方法**: 配置检出 `autovacuum[[:space:]]*=[[:space:]]*off` → warn。
- **对应门禁**: fw_pgsql_autovacuum(warn)

### 规律：必须用连接池（PgBouncer/应用池），禁止每请求新建连接
- **适用版本**: 全版本
- **规律**: PG 连接 = 一个后端进程（fork 成本 + 每连接数 MB 内存），直连上限数百即抖动。应用必须走池：Java HikariCP、或外部 PgBouncer（transaction 模式注意 prepared statement / session 级特性受限）。`maximumPoolSize` 按 `core*2 + spindle` 量级收敛，不是越大越好。
- **违反后果**: 每请求新建连接 → 建连 RT 叠加 + 后端进程爆内存 → `FATAL: sorry, too many clients`。
- **验证方法**: 配置检出 PG 数据源 URL 但全部配置无 `hikari|maximumPoolSize|pgbouncer|pool_size|pooling` → warn。
- **对应门禁**: fw_pgsql_conn_pool(warn)

### 规律：隔离级别须按需显式，SERIALIZABLE 慎用（默认 RC 起步）
- **适用版本**: 全版本
- **规律**: PG 默认 READ COMMITTED（语句级快照）；REPEATABLE READ 在 PG 是快照隔离（无幻读，与 MySQL 语义不同）；SERIALIZABLE（SSI）最严但有谓词锁开销与序列化失败重试义务（SQLSTATE 40001）。写冲突少的业务保持 RC；显式升 SERIALIZABLE 必须配套重试逻辑与压测依据。
- **违反后果**: 无脑 SERIALIZABLE → 40001 未捕获重试 → 请求直接报错；或误以为 PG RR 与 MySQL RR 相同而踩语义差。
- **验证方法**: 检出 `ISOLATION LEVEL SERIALIZABLE` → warn 核对重试逻辑与必要性。
- **对应门禁**: fw_pgsql_isolation(warn)

### 规律：索引类型须按场景选型（B-Tree/GIN/GiST/BRIN），模糊检索用 pg_trgm
- **适用版本**: 全版本
- **规律**: 默认 B-Tree 覆盖等值/范围；JSONB/数组/全文用 GIN；几何/范围重叠用 GiST；超大时序表按物理有序列用 BRIN（体积千分之一）。`LIKE '%abc%'` 双侧通配只有 `pg_trgm` 扩展的 GIN/GiST 索引能加速，否则全表扫。
- **违反后果**: 错选/不建索引类型 → 检索全表扫；BRIN 该用未用 → 时序大表索引占盘超数据。
- **验证方法**: 查询检出双侧通配 `LIKE '%...%'` 但仓库无任何 `pg_trgm` 引用 → warn。
- **对应门禁**: fw_pgsql_index_type(warn)

### 规律：日志/事件/历史类大表必须分区（PARTITION BY RANGE）
- **适用版本**: PG 10+（声明式分区）
- **规律**: 表名含 log/history/event 的追加写大表，不分区则清理只能 DELETE（产生死元组再 VACUUM，恶性循环），分区后可 `DROP PARTITION` 秒级归档。按月/周 RANGE 分区；分区键必须出现在高频 WHERE 中以享受分区裁剪。
- **违反后果**: 亿级单表 → 索引膨胀、VACUUM 跑不完、归档靠 DELETE 拖垮 IO。
- **验证方法**: `CREATE TABLE` 表名含 log/history/event 但同文件无 `PARTITION BY` → warn 核对数据量级。
- **对应门禁**: fw_pgsql_partition(warn)

### 规律：UPDATE/DELETE 必须带 WHERE，无条件 DML 禁上线
- **适用版本**: 全版本
- **规律**: 单行 UPDATE/DELETE 语句缺 WHERE = 全表改写/清空，且每行产生死元组 + 全程行锁。批量变更须 `WHERE` 限定范围并分批（每批 ≤1 万行 + 间歇），或 `TRUNCATE`（清表场景，DDL 级无死元组）。
- **违反后果**: 漏 WHERE → 全表数据损毁（生产事故第一高发）；长锁 + 膨胀。
- **验证方法**: 单行 `UPDATE ... SET ...;` / `DELETE FROM ...;` 语句不含 WHERE → fail。
- **对应门禁**: fw_pgsql_dml_where(fail)

### 规律：批量导入用 COPY，禁止单文件成百上千条 INSERT
- **适用版本**: 全版本
- **规律**: 逐条 INSERT 每条一次解析/规划/WAL 刷盘路径；`COPY t FROM STDIN` 走批量协议快 10~100 倍。迁移/种子脚本中成批 INSERT（≥50 条）应改 COPY 或多行 `INSERT ... VALUES (...),(...)`（单语句 ≤1000 行分批）。批量期间可临时 `DROP INDEX` 后重建。
- **违反后果**: 百万行导入从分钟级变小时级；WAL 洪峰触发主从延迟。
- **验证方法**: 单 .sql 文件 `INSERT INTO` 行数 ≥50 → warn 建议 COPY。
- **对应门禁**: fw_pgsql_copy_vs_insert(warn)

### 规律：序列必须设 CACHE（默认 1 高并发下是瓶颈）
- **适用版本**: 全版本
- **规律**: `CREATE SEQUENCE` 默认 `CACHE 1`，每次 `nextval` 都写 WAL 并更新序列页，高并发取号成热点。设 `CACHE 50`~`1000`（实例崩溃会丢弃缓存段，产生间隙——业务须容忍）。IDENTITY 列同理（底层序列可调 CACHE）。
- **违反后果**: 高并发 INSERT 的 nextval 锁竞争 → 写 TPS 天花板。
- **验证方法**: `CREATE SEQUENCE` 行显式 `CACHE 1` → warn；未标 CACHE 提示人工核对（本门禁只抓显式 CACHE 1）。
- **对应门禁**: fw_pgsql_seq_cache(warn)

### 规律：约束必须显式命名（CONSTRAINT name），禁用匿名系统名
- **适用版本**: 全版本
- **规律**: 匿名约束由系统生成 `t_pkey`/`t_col_fkey` 类名字，多环境重建后名字漂移，迁移脚本按名 DROP/ALTER 约束时跨环境失败。建表时 `CONSTRAINT pk_users PRIMARY KEY (id)` / `CONSTRAINT fk_orders_user FOREIGN KEY (user_id) REFERENCES users(id)` 显式命名。
- **违反后果**: 迁移脚本在测试环境能跑、生产报"constraint does not exist"。
- **验证方法**: CREATE TABLE 块内含 `REFERENCES` 但全文无 `CONSTRAINT` 关键字 → warn。
- **对应门禁**: fw_pgsql_constraint_naming(warn)

### 规律：统计信息须随大批量变更更新（ANALYZE / default_statistics_target）
- **适用版本**: 全版本
- **规律**: 优化器依赖 pg_statistic 直方图估值；大批量导入/删除后统计信息过期 → 错估行数 → 选错执行计划（nested-loop 用在亿行表上）。autovacuum 的 analyze 有滞后，批量变更后手工 `ANALYZE t`；倾斜列调 `ALTER TABLE t ALTER COLUMN c SET STATISTICS 1000`。
- **违反后果**: 计划突变 → 接口 RT 突增 10~100 倍且难以复现。
- **验证方法**: 人工检查（批量变更 Runbook 是否含 ANALYZE 步骤；`EXPLAIN (ANALYZE, BUFFERS)` 核对估算行数 vs 实际行数偏差）。
- **对应门禁**: 人工检查

## §5 跨框架交互规则

| 交互对 | 规则 | 理由 |
|-------|------|------|
| postgresql × spring-boot | HikariCP maxPoolSize 与 PgBouncer pool_size 须联动收敛；maxLifetime < server idle 超时 | 池 oversized → PG 后端进程爆；连接被服务端先回收 → 应用拿死连接 |
| postgresql × mybatis | 批量 insert 用 ExecutorType.BATCH 或 unnest 数组，JSONB 列须注册 JsonbTypeHandler | 逐条 insert 失去 PG 批量优势；jsonb 默认映射 String 丢类型 |
| postgresql × flyway/liquibase | 迁移脚本 DDL 须在事务内（PG 支持事务性 DDL），但 CREATE INDEX CONCURRENTLY 必须非事务单独文件 | CONCURRENTLY 在事务块内直接报错，混排导致整批迁移失败 |
| postgresql × elasticsearch | JSONB 全文/模糊检索超出 pg_trgm 能力时下沉 ES，PG 保事务源 | 双侧通配 + 相关性排序不是 PG 主战场 |

## §6 版本陷阱速查

| 版本 | 变化 | 影响 |
|------|------|------|
| PostgreSQL 10 | 声明式分区引入；IDENTITY 列引入；SERIAL 标记为兼容遗留 | 10 前分区靠继承+触发器，规则按声明式分区陈述 |
| PostgreSQL 12 | REINDEX CONCURRENTLY；generated column | 老版本索引重建锁写 |
| PostgreSQL 14 | 2026-11-12 EOL（调研时点仍在支持窗口尾期） | 存量 14 项目须提示半年内升级 |
| PostgreSQL 15 | MERGE 语句引入 |  upsert 场景可用 MERGE 替代 ON CONFLICT（差异语义须核对） |
| PostgreSQL 17 | incremental backup、JSON_TABLE | 备份策略可去 pg_basebackup 全量依赖 |
| PostgreSQL 18 | 异步 IO（AIO）默认生效；虚拟生成列默认 | IO 特征变化，云盘/本地盘压测基线须重测 |
| PostgreSQL 19 | Beta 阶段（19 Beta 2，2026-07-16），未 GA | 待验证：19 GA 后新特性与规律适配须复核 |
