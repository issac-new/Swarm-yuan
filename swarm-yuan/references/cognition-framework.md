## §1 第一层：认知递进

**六阶链**：①概念定义 → ②结构 → ③空间 → ④三者映射 → ⑤认知规律 → ⑥处理关系

**六维动力学**：速度（变更文件数）/ 聚散（服务/组件数）/ 趋势（依赖深度）/ 强度（fan-in）/ 能耗（文件行数）/ 累积量（TODO 累积）

**门禁**：`--cognition` ①-⑥ 体检，输出五层认知基底总分

**核心理念**：呈现递进的关系，而非仅关注计算。每个门禁的 fail/warn 背后是一条关系规律。

## §2 第二层：思维语言

**三元演化**：提高认识 → 重塑价值 → 驱动方法 → 指导实践 → 矛盾反馈（循环）

**三导向**：问题导向（spec §1.1 现状溯因）/ 目标导向（spec §1.2 SMART）/ 结果导向（spec §14 交付衰减）

**七推理**：归纳 / 演绎 / 溯因 / 类比 / 分解 / 假设检验 / 辩证（每推理在 workflow 节点有落点）

**7×7 双循环**：7 阶段（界定→分解→优先→分析→关键分析→综合→实施）× 7 方向（需求/设计/代码/测试/部署/运维/管理）

**门禁**：spec §14 交付衰减 + §15 蓝图任务；`--cognition` 检查三导向段

## §3 第五层：辩证认知

**本质与现象**：本质=构成要素的内在联系（第一层 ①-⑥）；现象=可观测信号（六维动力学）

**7 对辩证关系**：

| 对 | 辩证关系 | 工程映射 | swarm-yuan 落点 |
|----|---------|---------|----------------|
| 1 | 内容↔形式 | 业务逻辑 vs 代码结构 | `--layer`、`--stable-diff` |
| 2 | 原因↔结果 | 架构决策 vs 系统行为 | spec §1.1 溯因、`--impact` |
| 3 | 必然↔偶然 | 技术债趋势 vs 何时崩塌 | `--link-depth`、`--adr` |
| 4 | 现实↔可能 | 当前系统 vs 蓝图目标 | spec §14/§15 |
| 5 | 实践↔认识 | 编码/测试 vs 探查/设计 | workflow 4-Phase |
| 6 | 真理↔谬误 | 正确决策 vs 反模式 | 第四层偏差防范 |
| 7 | 绝对↔相对真理 | 当前最优 vs 终极最优 | 五层认知基底本身 |

**矛盾分析法**：每个 workflow 节点须识别主要矛盾 + 矛盾的主要方面。

**门禁**：spec §17 辩证映射分析；`--cognition` 第五层检查 reference-manual 含辩证映射表 + 7 对关系 ≥3 对

## §5 ECC v2.0 认知扩展（council + recursive-decision-ledger）

### Council（四声音议事会）

ECC 的 `council` 在模糊 go/no-go 决策时召集**四个独立声音**：

| 声音 | 角色 | 关注点 |
|------|------|--------|
| **Advocate** | 支持者 | 论证"做"的理由 |
| **Skeptic** | 怀疑者 | 论证"不做"的理由 |
| **Pragmatist** | 务实者 | 论证"如何做"的可行性 |
| **Visionary** | 远见者 | 论证"长期影响" |

**结构化分歧**：四个声音**独立**输出观点，不互相参考（避免 groupthink），然后综合。

**在目标技能中的落地：**
- 高复杂度决策（如"是否重构整个模块"）可引用 council 模式：派发 4 个 subagent 分别扮演 4 个角色
- 综合 4 个角色的观点后，由用户做最终决策（AI 不替代用户决策）

### Recursive Decision Ledger（递归决策账本）

ECC 的 `recursive-decision-ledger` 记录**决策过程**而非仅决策结果：

- 每个决策记录：选项 / 理由 / 预期结果 / 实际结果 / 偏差分析
- 递归展开：若决策导致子决策，递归记录
- 局部最优探索：记录"为什么选择局部最优而非全局最优"的理由

**在目标技能中的落地：**
- `.swarm-yuan/decisions.md` 可扩展为递归格式：每个决策带子决策列表
- 复盘时可追溯"为什么当时这样决定"（避免事后诸葛亮）

### 意图驱动开发（intent-driven-development）

ECC 的 `intent-driven-development` 将模糊/高影响变更转化为**可验证的验收标准**：

- 在实现前，将"模糊需求"转化为"可验证的 acceptance criteria"
- 每个 criterion 须有明确的成功/失败判定
- 实现后验证每个 criterion 是否满足

**在目标技能中的落地：**
- spec 阶段的"验收标准"段可引用此模式：每个需求须有可验证的 criterion
- 实现后验证 criterion（而非"以为实现了"）
