# 逻辑剃刀对抗审查方法论 (Logic Razor Adversarial Review)

本文件是 swarm-yuan 第三层"认知辩证"的**审查方法论 reference**，供生成的目标技能 check 段引用。核心理念：**审查者不得全盘肯定——即使方案看似无懈可击，也须挑出至少 10% 严谨性瑕疵。**

> 本文件为方法论 reference，不依赖任何具体项目内容。
> **执法姿态：纯阅读方法论，无门禁执法点**——precheck/verifier 均不校验本文件的执行（`--cognition` 仅对目标产物做关键词计分，不构成对本文件的执法）；"至少 10% 瑕疵"铁律与 6 模块审查报告是 AI 阅读后的自律要求。与 gsd abstain 原则的冲突裁决见文末「razor↔abstain 裁决条款」。

## 审查流程：两步法

### Step 1：隐性深度解剖（审查者在后台推演，不输出）

1. **拆解**：提取被审查对象的主张(Claim)、数据(Data)、隐含假设(Warrant)，区分客观事实与主观价值判断
2. **扫描**：对照谬误图谱（见下文）扫描语境/定义/逻辑/证据四层陷阱
3. **选型**：从分析框架（图尔敏/AREI/第一性原理/需根解损/金字塔）选取反击武器
4. **推演**：寻找反例、控制变量、归谬法推演极端场景

### Step 2：降维打击输出（审查报告格式）

审查报告须含以下 6 模块，缺一不可：

| 模块 | 内容 | 输出要求 |
|------|------|---------|
| 🎯 观点镜像 | 一句话复述被审查对象的核心论点与前提 | 确认无稻草人（不曲解原意） |
| 💥 核心定调 | 一句精炼话指出致命伤 | 不得用"各有各的道理"等和稀泥表述 |
| 🩻 逻辑病理学诊断 | 隐含假设审查 + 谬误确诊（1-3 种） | 须引用谬误图谱中的具体谬误名 |
| 🗡️ 降维反驳 | 引经据典+数据/思维实验，指出证据链断裂 | 须提供至少 1 个反例或极端场景 |
| 💡 建设性重构 | 跳出当前框架提供可验证改进方向 | 不得仅批评不建设 |
| ⚖️ 灵魂拷问 | 将举证责任踢回作者 | 至少 1 个待作者回答的问题 |

## 逻辑谬误图谱（四类）

审查时对照诊断，确诊须引用具体谬误名：

### 形式与语言类
| 谬误 | 一句话 | 工程场景 |
|------|--------|---------|
| 肯定后件 | 若 A 则 B，有 B 故 A | "测试通过故实现正确"（实现正确可能有其他原因） |
| 偷换概念 | 同一词前后含义不同 | "敏捷"从原则偷换成"不写文档" |
| 挪动门柱 | 达标后提高标准 | 代码审查不断加新要求 |
| 套套逻辑 | A 因为 A | "稳定因为它稳定" |
| 假精确 | 无意义的高精度 | "性能提升 73.45%"（基准不明） |
| 断章取义 | 截取部分语境 | 只引用支持的 ADR 条款 |

### 来源与人身类
| 谬误 | 一句话 | 工程场景 |
|------|--------|---------|
| 稻草人 | 曲解对方再反驳 | 把"优先复用"曲解成"禁止新建"再反对 |
| 红鲱鱼 | 转移话题 | 讨论安全漏洞时转到性能 |
| 匿名权威 | 引用不明来源 | "业界都这么做"（无具体案例） |
| 诉诸人身 | 攻击人非观点 | "这代码是实习生写的" |
| 诉诸传统 | 一直这样做故对 | "老系统就这样" |
| 诉诸自然 | 自然即对 | "原生实现比框架好" |

### 情感与归纳类
| 谬误 | 一句话 | 工程场景 |
|------|--------|---------|
| 诉诸恐惧 | 制造恐慌 | "不上这个方案会出大事故" |
| 转移举证责任 | 让对方证伪 | "你说没问题，证明给我看" |
| 偏差样本 | 非代表性样本 | "我测了 3 次都通过"（边界未测） |
| 德州神枪手 | 先开枪再画靶 | 先写代码再编需求 |
| 轶事证据 | 个例当规律 | "上次这样没出事" |
| 合成谬误 | 部分对故整体对 | "每个模块 tested 故系统 tested" |
| 不当类比 | 表面相似实质不同 | "微服务像乐高积木"（忽略了分布式复杂性） |
| 涅槃谬误 | 不完美故无用 | "有 bug 故不如不测" |

### 因果与预设类
| 谬误 | 一句话 | 工程场景 |
|------|--------|---------|
| 相关不蕴涵因果 | A 与 B 同时故 A 致 B | "部署后出 bug 故部署致 bug" |
| 后此谬误 | B 在 A 后故 A 致 B | "改代码后崩故改代码致崩" |
| 因果倒置 | 倒果为因 | "高耦合因共享库"（实为共享库致高耦合） |
| 单因谬误 | 归于单一原因 | "延期因某开发慢"（忽略需求变更） |
| 滑坡谬误 | 小步推到大灾难 | "加个 if 最终变意大利面" |
| 乞题 | 前提即结论 | "稳定单元不可改因为它是稳定的" |
| 假两难 | 只给两个选项 | "要么重构要么烂掉" |
| 混淆充要 | 充分当必要 | "有测试故安全"（测试非安全充分条件） |

## 分析框架武器库

审查者按需选取：

| 框架 | 适用场景 | 核心问题 |
|------|---------|---------|
| 图尔敏模型 | 论证结构分析 | 主张-数据-担保-支撑-限定-反驳六要素是否完整 |
| AREI | 简短论证 | Assertion-Reasoning-Evidence-Implication |
| 第一性原理 | 深挖假设 | 剥到不可再剥的物理约束是什么 |
| 需根解损 | 方案论证 | 需求-根因-解决-损益是否对齐 |
| 金字塔原理 | 结构化输出 | 结论先行、以上统下、归类分组、逻辑递进 |

## 与既有审查方法论的协同

| 审查维度 | 来源 | 关注点 |
|---------|------|--------|
| 5 维度（正确性/安全/性能/可维护/测试覆盖） | gstack/OCR（review-methodology.md） | 代码质量 |
| AUTO-FIX vs ASK | gstack | 处置策略 |
| **逻辑剃刀 6 步** | **本文件** | **论证质量（主张-证据-假设是否成立）** |
| goal-backward | gsd-core | 目标达成（任务完成≠目标达成） |

> 逻辑剃刀补充了既有审查缺失的"论证质量"维度——代码可能功能正确，但支撑它的设计决策论证可能有谬误。

## 目标技能落地要求

1. reference-manual.md 须含"逻辑谬误图谱"段（从本文件四类裁剪项目相关谬误）

## razor↔abstain 裁决条款（与 gsd-patterns.md 互引）

**冲突**：本文件铁律"审查者不得全盘肯定——即使方案看似无懈可击，也须挑出至少 10% 严谨性瑕疵"（见文首与上文落地要求第 4 条）与 gsd-core honest verifier 原则"spec 信息不足时**弃权**（`abstain: insufficient_spec`）而非猜测"（`references/review-methodology.md` Honest Verifier Abstain 段；`references/gsd-patterns.md` Goal-Backward 段）不可同时为真——一个强制产出批评，一个禁止无据产出。

**裁决（按轮次证据完备度分治）**：

- **razor 适用于证据充分的审查轮次**：被审查对象材料完整、可推演、可反证时，禁止以"信息不足"为由全盘肯定或弃权——仍须挑出 ≥10% 严谨性瑕疵。
- **abstain 适用于证据不足的探查轮次**：spec/材料不足以推断 backstop truth 时，禁止编造瑕疵凑满 10%——输出 `abstain: insufficient_spec` + 待补信息清单，补充后重新审查。弃权 ≠ 通过（`--review` 中计为"需人工确认"，非 pass 非 fail）。
- **判据 = 证据完备度 checklist**（全部满足 = 审查轮次；任一不满足 = 探查轮次，abstain 输出须指明缺哪条）：
  - [ ] 主张明确：核心论点可一句话复述（观点镜像无歧义）
  - [ ] 数据可得：关键证据可追溯（文件:行 或命令输出）
  - [ ] 假设可识别：隐含假设（Warrant）可显式提取
  - [ ] 反例可构造：材料充分到能推演极端场景/边界反例
  - [ ] 结论可复验：第三方按同样材料可复现审查结论
- **两条严禁**：证据不足强行挑 10% 瑕疵 = 编造（违反 honest verifier）；证据充分却弃权 = 逃避对抗审查（违反 razor 铁律）。
